# Assumptions

## Active Circle Persistence
Active circle is stored as a ForeignKey on the User model (`active_circle`). This is the simplest clean solution:
- Persists across sessions (unlike session-based storage)
- Automatically null when the circle is deleted (`on_delete=SET_NULL`)
- Single database query to read/write

## Authentication
- Django built-in auth with session cookies
- CSRF tokens delivered via cookie (`CSRF_COOKIE_HTTPONLY = False`) so the React SPA can read them
- Login and register endpoints use Django's `authenticate()` and `login()` functions
- Email is the unique identifier (USERNAME_FIELD)

## Architecture
- React frontend consumes Django private API at `/api/private/`
- Same domain via Kubernetes ingress (frontend port 3000, backend port 8001)
- Cookies work seamlessly since both are same-origin from the browser's perspective
- Django ASGI app served by uvicorn (compatible with existing supervisor config)

## Invitation Links
- Single-use in V1 (once accepted, the invite is marked as used and deactivated)
- 7-day expiry by default
- Token is a random UUID
- Invitation acceptance adds user to circle as MEMBER role

## Event Signup Flow
- User requests signup → status = PENDING
- Circle admin approves → status = APPROVED
- Circle admin rejects → status = REJECTED
- One signup per user per event (unique constraint)

## Role-View Dropdown
- Always shows "User" view
- Shows "Admin" view if user is CIRCLE_ADMIN in at least one circle
- Shows "Site Manager" view if user.is_site_manager is True
- All three visible if user is both a circle admin and site manager

## Database
- PostgreSQL is used instead of the default MongoDB in the environment
- Added as a supervisor-managed process for auto-restart
- Migrations are auto-generated by Django
